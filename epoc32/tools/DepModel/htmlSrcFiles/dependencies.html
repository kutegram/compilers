DepModel Static Dependency Modelling Tool

The pages referenced by this dependency modelling tool provide all the information developers need on static dependencies in Symbian OS, giving static dependency information on every executable (EXE or DLL), component and subsystem in Symbian OS.

For top-level access to this information use one of the index pages at SUBSYSTEMS INDEX   COMPONENTS INDEX   EXECUTABLES INDEX . These index pages provide easy access to the information on the other pages. Hyperlinks to each of these index pages are provided at the top of almost every other page in the tool, as shown on this page itself.

Using these pages does not require technical expertise. You can use this service to:

  • Find textual dependency information on any executable/component/subsystem.
  • Find the dependency diagram for any executable/component/subsystem, with hotspots enabling easy navigation between pages.
  • Find out why any executable/component/subsystem depends on any others (imports and exports).
  • Find a list of all executables sorted by subsystem and component with a brief description explaining the executable's purpose.
  • Drill down to the precise function calls that cause an executable/component/subsystem to depend on another.
  • Find out the dependency layering between all executables (or components or subsystems).
  • Find a one-line description of most executables and components, size information, build files, etc.

Key use cases for using this service can be found here.

The scripts that generated these pages can be used to generate similar pages based on any release or on a licensee ROM.

Further general information

Using SUBSYSTEMS INDEX   COMPONENTS INDEX   EXECUTABLES INDEX index pages you can follow a link to the dependency information page for each item on that page, to see the details of these dependencies both in textual and graphical formats. Here the word 'item' indicates either an Executable, Component or Subsystem depending on the specific index being used. As well as providing an alphabetic listing of the items, these index pages show how the items are layered by dependency. Items with circular dependencies are treated as being at the same level.

In the dependency trees, a trailing '+' after the name indicates a collapsed item that is expanded elsewhere in the same tree at the position furthest to the right.

  • View dependencies between executable ROM Files
    These dependencies are found by direct analysis of the binary files using the PETRAN tool. These dependencies will correspond to the library declarations specified in each SymbianOS executable's MAKE file (*.mmp).

    Each executable is in exactly one subsystem/component.

    The executables view lists all DLLs and EXEs layered by dependency, with further links to dependency trees and graphs for each executable.

    The icon after a ROM file's name is a link to detailed information, giving the fully qualified C++ function names of all imports and exports used by the current page's file. E.g. when viewing agnmodel.dll's page, clicking the icon in "agsvexe.exe", which uses agnmodel.dll, links to the list of functions that agsvexe.exe imports from agnmodel.dll. Similarly clicking the icon in "apparc.dll", which is used by agnmodel.dll, links to the list of functions that agnmodel.dll imports from apparc.dll.

  • View dependencies between Components
    A component is a collection of releasables from one source tree with strong interdependencies between its ROM files. Many components make up a ROM. Each subsystem/component belongs in exactly one subsystem. The ROM files in a component are specified in the component's bld.inf files (and the location of the bld.inf files for GT is specified, in turn, in the GT.txt file used in the mainline build process).

    This view simply treats componentA as dependent on componentB if any executable in componentA depends on any in componentB, so it is fully determined by the executable ROM File dependency.

  • View dependencies between Subsystems
    A subsystem is intended to be a logical grouping of components. In some cases this grouping is currently not logical. As a result of this dependency analysis, it is planned that components will be reassigned to other or to new Subsystems where they more logically and architecturally fit.

    This view treats subsystemA as dependendent on subsystemB if any component in subsystemA depends on any in subsystemB, so it is also fully determined by the executable dependency.
